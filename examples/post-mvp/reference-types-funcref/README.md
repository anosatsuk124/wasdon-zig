# Post-MVP example: reference-types `funcref` + `call_indirect` table index

This example exercises the parser-level reference-types relaxations the
translator needs to accept output from any modern wabt / LLVM
toolchain:

1. `funcref` (value-type byte `0x70`) appearing in a table type.
2. The `call_indirect` instruction's *second* immediate — the table
   index — encoded as a uleb128 (instead of the MVP "reserved 0x00").

The module declares:

- A trivial signature `$sig = (param i32) (result i32)`.
- A concrete callee `$square` matching `$sig`.
- A `funcref` table `$tab` of size 1, populated via an active element
  segment containing `ref.func $square`.
- A caller `$call_sq` that pushes its `i32` argument and `i32.const 0`,
  then `call_indirect $tab (type $sig)`.

In the produced binary `call_indirect` is encoded as `11 00 00`
(opcode `0x11`, typeidx uleb=`0x00`, table_idx uleb=`0x00`). The
*value* is byte-for-byte identical to an MVP encoding, but the parser
must now decode that second byte as a uleb128 rather than rejecting any
non-`0x00` value. Forcing a multi-byte LEB at this position would
require `table_idx >= 128`, which in turn requires importing or
declaring 128+ tables — well outside the scope of a single example.
A direct multi-byte LEB test lives instead in
`src/wasm/instruction.zig`'s unit tests (synthesised module bytes), so
this example only covers the on-the-wire shape a real toolchain emits.

## Build

```sh
wat2wasm example.wat -o example.wasm
```

(In wabt 1.0.39 the reference-types and bulk-memory proposals are on by
default. WAT note: write the table immediate *before* the type clause:
`call_indirect $tab (type $sig)` — wabt rejects the reverse order.)

## Verify

`wasm-objdump -x example.wasm` should show:

```
Table[1]:
 - table[0] type=funcref initial=1 max=1
Elem[1]:
 - segment[0] flags=0 table=0 count=1 - init i32=0
  - elem[0] = ref.func:0
```

`wasm-objdump -d example.wasm` should disassemble `$call_sq` to:

```
local.get 0           ;; 20 00
i32.const 0           ;; 41 00
call_indirect 0 (type 0)   ;; 11 00 00
end                   ;; 0b
```

The two trailing `0x00` bytes after the `0x11` opcode are
`typeidx=uleb(0)` and `table_idx=uleb(0)` respectively.

## Translate

```sh
zig build run -- examples/post-mvp/reference-types-funcref/example.wasm
```

Expected markers in the emitted Udon Assembly:

- `$call_sq` lowers to a `call_indirect` dispatch equivalent to the
  MVP path (the single supported table is `$tab` at index 0).
- The `funcref` table type is accepted by the decoder but does NOT
  appear as a declared field on the UdonBehaviour (table state is
  encoded in the dispatch table generated by the translator).
- No `ANNOTATION __unsupported__` line for `call_indirect`.
- A WAT variant with `table_idx > 0` (e.g. two declared tables) MUST
  fail with `MultiTableNotYetSupported` — verified in the translator
  unit tests, not in this example.

See `docs/spec_call_return_conversion.md` §"call_indirect with explicit
table index (reference-types)" for the dispatch lowering and the
single-table restriction.
